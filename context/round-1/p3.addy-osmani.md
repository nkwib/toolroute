# addy-osmani on "ToolRoute"

**Score:** 6/10
**Would I share this?** maybe — the problem is real and the scope is honest, but I'd want to see adoption numbers before amplifying

## Strengths

- The core insight maps cleanly to what I've been calling the "harness problem": the gap between model capability and observed output is usually a structural failure in how tools are exposed, not a model failure. ToolRoute addresses that gap at the right layer — compile time, not system prompt.
- Scope discipline is genuinely rare here. 500 LOC of types plus a 50-line runtime guard is a surface area a senior engineer can audit in an afternoon. In practice, small auditable packages get adopted; monolithic agent frameworks get evaluated and then quietly abandoned.
- Narrowing the permitted tool union per turn is the right UX signal for teams that don't want to learn XState. It trades expressiveness for immediacy — that's a legitimate trade for a solo dev building a sales-research bot.

## Concerns

- The Vercel AI SDK dependency is a single-point-of-lock-in risk. In practice, SDK tool shapes change fast — `streamText`, `generateText`, tool interop — and a narrowing type built on top of a moving target accumulates drift. Six months from now, a minor Vercel AI SDK version bump could silently break the narrowing without a compile error, which is precisely the kind of failure this tool promises to prevent.
- The adoption story is thin without a clear "who discovers this" path. The Theo-tier audience is real, but they discover things through demos and numbers, not through OSS releases alone. There's no viral mechanic here — no shareable trace, no diff, no before/after that travels on its own.
- `nextAllowed` as a const tuple is ergonomically fragile under refactors. Renaming a tool means updating every `nextAllowed` reference that mentions it — not enforced by the type system unless tool names are derived from the same const source. In practice this becomes a maintenance burden that nudges teams back toward "just write it in the system prompt."

## Pivots

- Derive tool names from the same const source of truth as `nextAllowed` entries — something like `createRouterFromTools([toolA, toolB])` where `toolA.name` is the only string literal in play. This eliminates the rename-drift problem and makes the type narrowing genuinely self-consistent.
- Ship one concrete vertical example — a three-tool code-review agent with a recorded terminal session showing the red squiggle when you try to call `commit` before `review` — and put it at the top of the README and docs page. Data-free OSS launches don't spread; a 30-second demo that shows the actual error does.

## Feasibility flags

- **Solo-shippable:** yes — 2 weeks is credible for 500 LOC of types + docs site; no backend surface
- **Legal/compliance:** clean — pure OSS types package, no data ingestion, no PII surface
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** partial — TS core is solid; Svelte docs site is fine; the Vercel AI SDK constraint means it sits outside the Anthropic SDK path unless the adapter story is explicit
- **OSS or paid:** OSS is correct here — a focused calling-card package; monetization should come from a downstream tool (AgentGraph, AgentAssert) not from ToolRoute itself
