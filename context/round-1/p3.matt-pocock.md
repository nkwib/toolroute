# matt-pocock on "ToolRoute"

**Score:** 8/10
**Would I share this?** yes — the per-turn union narrowing trick is exactly the kind of "types doing real work" idea I'd turn into a Total TypeScript post immediately.

## Strengths

- The `nextAllowed: ['toolB', 'toolC'] as const` pattern is genuinely novel in the agent tooling space. Collocating the legality graph with the tool definition itself — rather than in a separate transition map — is a teachable, memorable API shape. That's rare.
- Per-turn union narrowing based on prior tool output is the most surprising type-level move in all three proposals. It's not obvious that TypeScript can even express "the tools object narrows on each step of the loop" until you see it done. That surprise is what makes it blog-worthy.
- Scope is honest. ~500 LOC of types plus 50-line runtime guard is a library you can fully understand in one sitting. That's the right size for a calling-card OSS project and makes it extremely teachable.

## Concerns

- The Vercel AI SDK `streamText` tool union narrowing may hit a real wall: the AI SDK hands `tools` as a static object to the model at call time, not dynamically per step. If the narrowing is purely a compile-time illusion — and the runtime still passes the full tool set to the model — then the guarantee is decorative, not operational. That's a DX lie I can't endorse.
- `defineTool` wraps existing AI SDK tool definitions, which means you're fighting the SDK's own type inference. Inference chains that cross an adapter boundary tend to produce horrific error messages. The teach-ability of the pattern depends entirely on error messages being intelligible when you break it — which is not guaranteed here.
- The `composeWithAgentAssert` helper coupling is dangled without being scoped. Either ToolRoute is a standalone primitive or it's the legality-graph module of a larger suite; leaving it as a teaser in the MVP adds surface area that confuses the story.

## Pivots

- Prove the runtime narrowing first, in a recorded demo, before writing types. Build the 50-line runtime guard that actually passes only the legal tool subset to `streamText` on each step, then layer the types on top so they match observable reality. If the runtime guard works, the types are honest. If it doesn't, the whole premise collapses — and you want to know that on day one, not after shipping.
- Drop the `composeWithAgentAssert` mention entirely from v1. Ship ToolRoute as a single, complete idea: `defineTool` + `createRouter` + a live playground showing the union shrink in the editor. The AgentAssert bridge is a v2 story; putting it in the MVP docs makes the scope look leaky.

## Feasibility flags

- **Solo-shippable:** yes — 2 weeks is plausible if the AI SDK adapter boundary cooperates; call it 3 to be safe
- **Legal/compliance:** clean — pure OSS types library, no data ingestion, no user content
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — TS core, Svelte playground, OpenAI/Anthropic adapters all in-stack
- **OSS or paid:** OSS is exactly right — this earns its value as a calling card and a teachable primitive; a paid tier here would feel extractive

## Quote (1 sentence in their voice)

> "Right — so the clever bit isn't the `nextAllowed` tuple, it's that on every turn the types are actively doing work to shrink the union, and if your runtime isn't doing the same thing then you've built the world's most elaborate lie detector for your own code."
