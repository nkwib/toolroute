# addy-osmani on "ToolRoute"

**Score:** 8/10
**Would I share this?** yes — the recorded demo is the thing; a terminal session showing a loud runtime rejection before the type layer even exists is exactly the kind of before/after artifact that lands with practitioners

## Strengths

- **Single source of truth closes the rename-drift gap in practice.** The `createRouterFromTools` design — where tool names are derived from the const array, not re-declared as strings in `nextAllowed` — is the right call. In practice, the failure mode for constraint systems like this is not the type system lying, it is the author updating one declaration and forgetting the string literals elsewhere. This revision removes that entire class of drift.
- **Runtime-first sequencing is the correct pedagogy.** Building and demoing the `strictMode` rejection before layering on types means the guarantee is grounded in observable behavior. Teams evaluating this can read the terminal session and trust it without understanding the type machinery. That separation matters for adoption across skill levels — the junior dev on the team can see it work without reading the inferred union.
- **Scope is now genuinely narrow.** Two weeks, ~550 LOC, one SDK target, one complete example agent. That is a scope I can point to as a realistic solo project. The decision to drop `composeWithAgentAssert` is correct — shipping one complete idea at this surface area beats shipping two half-ideas.

## Concerns

- **The live playground on the docs site is scope risk.** A Svelte/Cloudflare Pages playground that reflects real SDK behavior requires keeping the playground's bundled SDK version in sync with whatever Vercel ships. In practice that maintenance burden is invisible at launch and painful six months later. The recorded terminal session demo is enough for v1 — the playground can wait for v2 when there is community demand to justify the upkeep.
- **`strictMode: false` warn path needs a decision on where the warning goes.** `console.warn` is fine for Node but gets swallowed in edge runtimes and Vercel's streaming context. If the warning silently disappears in the environment where most users will run this (Vercel Edge Functions), the non-strict mode is a false sense of safety. The docs need an explicit callout, and the implementation should probably emit to `stderr` with a structured prefix that survives log aggregation.
- **No data on how often real Vercel AI SDK users hit tool-ordering bugs.** The problem statement is credible from intuition, but the README comparison table vs XState and LangGraph.js would be stronger with even one cited Discord thread, GitHub issue, or community post showing this pain point. Without it, this reads as a well-engineered solution to a problem the author personally hit — which may be real but does not yet signal broad demand.

## Pivots

- **Cut the live playground from v1 scope, replace with a static diff in the README.** Ship a side-by-side code block: the raw Vercel AI SDK tool definition on the left, the ToolRoute-wrapped version on the right, with a callout showing the narrowed `nextAllowed` union. This is zero-maintenance and communicates the same delta as a playground — and the recorded terminal session already handles the dynamic demo use case.
- **Add a `strict` runtime check that detects edge-runtime environments and surfaces a one-time warning at initialization** — something like `ToolRoute detected an edge runtime: strictMode rejections will throw, but console.warn is suppressed; pipe your runtime logs to capture violations`. This is five lines of code and directly addresses the warn-path gap; it also becomes a teaching moment in the docs about how to handle structured logging in edge contexts.

## Feasibility flags

- **Solo-shippable:** yes — 2 weeks is credible at ~550 LOC; the recorded demo is the most time-consuming artifact and should be scripted first
- **Legal/compliance:** clean — pure OSS wrapping a public SDK, no data collection, no hosted ingest
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — TS core, Svelte for the docs page, Vercel AI SDK is the explicit target; Anthropic SDK is adjacent scope for v2
- **OSS or paid:** OSS is right for v1 — this earns trust in the ecosystem before any monetization surface makes sense; a paid hosted-routing-audit layer is a plausible v2 if adoption materializes
