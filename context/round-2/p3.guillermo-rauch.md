# guillermo-rauch on "ToolRoute"

**Score:** 9/10
**Would I share this?** yes — the router-as-primitive metaphor finally landed; this is the kind of DX abstraction I'd want shipping on Vercel's AI SDK surface

## Strengths

- `createRouterFromTools` is the right mental model. Routing is the abstraction developers already understand from HTTP — the jump from "route per URL" to "route per tool call" is a zero-friction conceptual transfer. That's rare.
- Dropping `composeWithAgentAssert` from v1 is exactly right. One idea, fully shipped, beats two ideas half-baked. The narrowing per prior tool-call is already a complete primitive — it didn't need a second composition API bolted to it.
- Runtime guard that throws loud with the offending tool name, the prior tool, and the legal next set is what separates a type toy from production infrastructure. That error message is the product.

## Concerns

- The Vercel AI SDK coupling is a real constraint. SDK internals change fast — if `streamText`'s tool-call shape shifts in a minor version, the narrowing breaks silently for users who don't pin. Needs an explicit peer-dep range and a CI test against the latest SDK drop.
- The live playground on Cloudflare Pages sounds light but it's actually a second cold start: someone has to build and maintain it. If the README example and the terminal recording are sharp enough, the playground is distraction, not value — ship it in v2.
- Tool names as the single source of truth from the const array is elegant, but renames across a live routing graph in a real codebase will surface confusion fast. A zero-friction rename story (or at minimum a warning in the guide) should be in the README on day one, not discovered in a GitHub issue.

## Pivots

- Kill the Cloudflare playground for v1. Put that energy into the terminal recording and a copy-paste-ready three-tool example in the README. The playground is a distraction that adds a second thing to maintain before you have any users.
- Add a `routerVersion` field to the emitted violation error — include the SDK version the router was compiled against. When SDK internals drift and narrowing breaks, this is the first thing support will ask for. Build the diagnostic surface in before the breakage happens.

## Feasibility flags

- **Solo-shippable:** yes — 2 weeks is credible; 500 LOC of types + a 50-line guard is the right scope
- **Legal/compliance:** clean — pure OSS wrapper around a public SDK, no data retention, no third-party calls
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — TS-native, Vercel AI SDK is the primary target, Svelte only for the playground surface
- **OSS or paid:** OSS is correct for v1; the router is a calling card, not a revenue vehicle — monetization is a hosted violation dashboard or a Pro tier on the playground, both documented as v2
