# matt-pocock on "ToolRoute"

**Score:** 9/10
**Would I share this?** yes — `createRouterFromTools` is a genuinely teachable single-source-of-truth pattern and the 50-line runtime guard gives the types something real to describe

## Strengths

- `createRouterFromTools([toolA, toolB])` deriving names from the const array itself is exactly the kind of move I'd write an article about — the rename-drift problem is killed structurally, not by convention, and that distinction is worth 1,000 words of docs
- Building the 50-line runtime guard first and recording the terminal rejection before a single type is written is the correct epistemology: types describe observable reality, not aspirational behavior; this proposal now says that out loud, which round 1 didn't
- The scope is tight enough to be teachable in a single workshop module — `defineTool`, `createRouterFromTools`, one three-tool example, done; no sprawl, no "and also here's our state machine abstraction"

## Concerns

- The Vercel AI SDK coupling is still the ceiling: if the SDK reshapes its tool call interface (which it has done mid-minor-version before), the narrowing breaks and the library reads as unmaintained overnight; the README acknowledgement helps but doesn't solve it
- `nextAllowed` as a const tuple on each tool definition is local reasoning — it means you can't see the full routing graph without reading every tool file; `createRouterFromTools` gives you the graph at composition time, but there's no way to inspect or visualize it without running code
- The playground on Cloudflare Pages adds a second cold start risk for a v1 — if the core OSS package doesn't get traction first, a live playground nobody visits becomes a maintenance drag; cut it from v1 and ship it only once the README has GitHub stars

## Pivots

- Expose a `printRouterGraph(router)` utility that emits a plain-text adjacency list to stdout — zero deps, two minutes to implement, makes the routing graph inspectable without running the agent; this directly patches the "can't see the full graph without reading every file" concern and becomes the hero of the README terminal recording
- Version-pin the Vercel AI SDK peer dependency to a minor range with an explicit `COMPATIBILITY.md` table from day one — not a semver promise, just a public changelog of which SDK versions were tested; this turns the coupling risk into a visible, maintained artifact rather than a silent assumption

## Feasibility flags

- **Solo-shippable:** yes — 2 weeks is credible; 500 LOC types + 50-line guard + one example + README is a realistic solo sprint; drop the Cloudflare playground to hold that estimate
- **Legal/compliance:** clean — pure OSS types + thin runtime, no data ingestion, no user content
- **Stack fit (TS / Svelte / Supabase / AWS / Stripe / OpenAI / Anthropic):** yes — TS-first, Vercel AI SDK is adjacent to the existing stack, Svelte playground is optional and already known
- **OSS or paid:** OSS is correct — this is a calling card; the monetizable surface is a hosted routing-graph visualizer or a paid "ToolRoute Pro" audit report, both clearly v2
